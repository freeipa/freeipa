# External ACME Server Integration with FreeIPA

## Overview

FreeIPA includes a built-in ACME server based on Dogtag's ACME implementation,
enabled via `ipa-acme-manage enable`. That server handles the common case:
ACME clients inside the IPA domain obtain short-lived TLS certificates signed
directly by Dogtag, with no additional infrastructure required.

Some deployments require a **third-party ACME server** instead. Examples
include organizations that already operate step-ca, Boulder, or a custom ACME
implementation and want to integrate it with the FreeIPA CA rather than deploy
a separate PKI. The reasons vary:

- The third-party server supports ACME extensions or challenge types not yet
  implemented in Dogtag ACME (e.g., ACME Renewal Information, device-attest
  challenges).
- Multiple ACME servers are deployed in different network segments, including
  segments that do not have continuous connectivity to the IPA CA server.
- An organization policy requires that the ACME server be operated and audited
  independently of the IPA infrastructure.
- The ACME server needs to issue certificates under a CA that chains to the
  IPA root but is managed by a different team.

This family of designs describes how a third-party ACME server integrates with
FreeIPA so that certificates it issues are:

- signed by a FreeIPA sub-CA and therefore trusted by all IPA-enrolled clients,
- issued only to IPA principals (hosts, services, and optionally users) that
  have been explicitly authorized by an IPA administrator,
- subject to the same certificate profile constraints (key usage, validity,
  allowed names) as certificates issued directly by IPA.

## Core Components

Regardless of which integration model is chosen, the following FreeIPA
components are involved.

### Dedicated Sub-CA

Each external ACME deployment is backed by its own IPA lightweight sub-CA,
created with `ipa ca-add --acme`. Multiple ACME-dedicated sub-CAs can coexist
in the same IPA deployment â€” for example, separate sub-CAs for different
network segments, partner organizations, or teams. Each external ACME server
instance is associated with exactly one sub-CA.

The `--acme` flag applies the `ipaACMECA` auxiliary objectClass to the sub-CA's
LDAP entry (`ipacaid`, `cn=cas,cn=ca,$SUFFIX`), marking it as managed by an
external ACME server. This marker:

- scopes `ipa ca-find --acme` to list only ACME-dedicated sub-CAs,
- is required before the sub-CA can be referenced in `ipa ca-acme-add` commands,
- is the anchor for CA ACLs that bind authorized principals to this sub-CA.

Sub-CA names should reflect their deployment purpose. Using a generic name like
`acme-external` is discouraged; a naming convention such as
`acme-<deployment-id>` (e.g. `acme-corp-us`, `acme-dmz-partner`,
`acme-factory-floor`) makes the intent clear and avoids collisions when
multiple deployments are added later.

Using a dedicated sub-CA per deployment means:

- All ACME-issued certificates from that deployment share a distinct issuer DN,
  making them identifiable in audit logs and revocation infrastructure.
- The sub-CA can carry a `nameConstraints` extension limiting the DNS names it
  may sign, providing a cryptographic guarantee that the ACME server cannot
  issue certificates for domains outside the IPA environment.
- The sub-CA can be revoked or replaced without affecting the IPA root CA,
  any other ACME deployment, or the built-in Dogtag ACME server.

In the RA proxy model the sub-CA key is generated by Dogtag and stays in the
IPA CA. In the HSM-backed model the sub-CA key is generated on the ACME
server's own HSM; IPA signs the sub-CA certificate from a submitted CSR and
never holds the private key (`externalKey: true` in the Dogtag authority
record).

### ACME Certificate Profile

A dedicated Dogtag certificate profile (`acmeExternalServerCert`) governs what
end-entity certificates the external ACME server may produce. The profile
enforces:

- Maximum validity of 90 days (the ACME standard, matching Let's Encrypt and
  the built-in Dogtag ACME profile).
- Key usage: `digitalSignature` and `keyEncipherment`, critical.
- Extended key usage: TLS server authentication and TLS client authentication.
- SAN passthrough from the CSR (IPA validates names against the requesting
  principal independently).
- Acceptable key algorithms and sizes (RSA 2048+ or ECDSA P-256/P-384/P-521).
- AIA OCSP and CRL Distribution Point extensions pointing to the IPA CA.
- Issued certificates are **not** stored in the IPA LDAP directory
  (`--store false`), avoiding directory bloat from high-volume short-lived
  certificates.

In the RA proxy model this profile is evaluated by Dogtag at signing time. In
the HSM-backed model the profile is applied locally by the ACME server itself;
Dogtag is not involved at end-entity issuance time.

### Sub-CA Signing Profile

When creating an ACME-dedicated sub-CA, the signing profile applied to the
sub-CA certificate itself differs by model:

- **RA proxy model**: `ipa ca-add` without `--csr-file` uses the `caCACert`
  profile. Dogtag generates the key pair and signs a self-referential sub-CA
  cert with no path length restriction.
- **HSM-backed model**: `ipa ca-add --csr-file` uses the `caExternalKeyCACert`
  profile (default when `csrData` is present). This profile enforces
  `pathLen=0` (the sub-CA cannot itself issue further sub-CAs), rejects RSA
  keys below 2048 bits, and copies extensions present in the submitted CSR
  (e.g. `nameConstraints`) verbatim into the issued certificate.

### External Account Binding (EAB)

EAB (RFC 8555 Â§7.3.4) is the ACME mechanism by which a server requires clients
to prove membership in an external account system before obtaining certificates.
In this integration, an EAB account binds an IPA principal (host, service, or
user) to a specific ACME-dedicated sub-CA. Because multiple ACME sub-CAs can
coexist, a principal may hold multiple EAB accounts â€” one per sub-CA it is
authorized to use.

EAB accounts are stored as standalone entries in a dedicated LDAP container,
`cn=acmeaccounts,cn=ca,$SUFFIX`, using the structural objectClass
`ipaACMEAccount`. Keeping accounts in their own container (rather than
embedding them in host or service entries) allows one principal to have
accounts on several ACME sub-CAs without schema conflicts. Each entry carries:

- `ipaACMEAccountID` â€” the Key Identifier (KID), a public random string used
  to look up the account. Globally unique; used as the entry's RDN.
- `ipaACMEHMACKey` â€” a 32-byte secret HMAC key. Readable only by the ACME
  server's service principal and IPA administrators; never returned to the
  principal that owns the account.
- `ipaACMESubCA` â€” the `cn` of the ACME-dedicated sub-CA this account is valid
  for. An external ACME server only accepts EAB credentials whose
  `ipaACMESubCA` matches its own configured sub-CA.
- `ipaACMEPrincipal` â€” the DN of the IPA host, service, or user this account
  belongs to.
- `ipaACMEEnabled` â€” boolean flag; accounts can be suspended without deletion.

EAB accounts are managed with `ipa ca-acme-*` commands, where the sub-CA
name is the primary object and the KID is the account identifier:

```bash
# Create an EAB account for a host or service under a specific sub-CA:
ipa ca-acme-add acme-corp-us --host  server1.ipa.example.com
  # â†’ prints KID and HMAC key once
ipa ca-acme-add acme-corp-us --service HTTP/app.ipa.example.com
  # â†’ prints KID and HMAC key once

# Rotate the HMAC key (KID is preserved):
ipa ca-acme-mod acme-corp-us <kid> --hmac-key
  # â†’ prints new HMAC key once; old key is immediately invalidated

# Show, list, and delete:
ipa ca-acme-show acme-corp-us <kid>
ipa ca-acme-find acme-corp-us
ipa ca-acme-del  acme-corp-us <kid>

# List all ACME-dedicated sub-CAs:
ipa ca-find --acme
```

The HMAC key is displayed once at creation time and cannot be retrieved
afterwards. If the key is lost or needs to be rotated, use `--hmac-key` with
`ca-acme-mod` to regenerate it in place. The KID (Key Identifier) is
preserved, so only the ACME client's HMAC secret needs to be updated â€” not
the account registration. If the KID must also change (e.g. suspected
credential compromise), use `ca-acme-del` followed by `ca-acme-add` to
create a fully fresh account.

### Access Control

Three layers of access control govern what the external ACME server may issue.

**EAB validation** â€” at ACME account registration time, the server verifies
the client's HMAC signature using the secret stored in IPA. A client that does
not hold a valid EAB credential for a recognized IPA principal cannot register.

**Identifier authorization** â€” when an ACME account requests a certificate,
the server checks that the requested identifier is an allowed name for the
EAB-bound IPA principal before issuing a challenge:

| Principal type | Allowed identifier types | Source of truth in IPA |
|---|---|---|
| `host/h.example.com` | DNS names | `fqdn` and `ipaHostAliases` on the host LDAP entry |
| `<svc>/h.example.com` | DNS names | FQDN of the host part of the service principal |
| User principal | Email (RFC 822) | `mail` attribute on the user LDAP entry |

In the RA proxy model IPA's `cert-request` command enforces the same rules
server-side at signing time. In the HSM-backed model identifier authorization
is enforced entirely by the ACME server's own adapter logic, with the
`nameConstraints` extension on the sub-CA certificate as a cryptographic
backstop enforced by TLS relying parties.

**CA ACL** â€” IPA's existing CA ACL mechanism (`ipa caacl-*`) enforces which
principals may use which certificate profile with which sub-CA. Each
ACME-dedicated sub-CA has its own CA ACL entry linking the set of permitted
principals (hosts, host groups, services) to that sub-CA and the
`acmeExternalServerCert` profile. In the RA proxy model requests that do not
match the CA ACL are rejected by IPA before Dogtag is involved. In the
HSM-backed model the ACME adapter enforces the equivalent check locally using
the EAB-bound principal's LDAP attributes.

### ACME Server Service Principal

The external ACME server authenticates to IPA as a Kerberos service principal
(`acme/acmeserver.ipa.example.com`), created with `ipa service-add`.

### RBAC Specification

The following tables specify every new permission, privilege, and role this
integration introduces. All new permissions are standard IPA-managed LDAP ACIs;
exact OID assignments follow IPA's registered arc and are determined during
implementation.

#### Permissions

EAB accounts live in the dedicated container `cn=acmeaccounts,cn=ca,$SUFFIX`,
so permissions target that subtree rather than the host or service object types.

| Permission | Target subtree | `targetattr` | `targetfilter` | Rights |
|---|---|---|---|---|
| Manage ACME Accounts | `cn=acmeaccounts,cn=ca,$SUFFIX` | `ipaACMEAccountID`, `ipaACMEHMACKey`, `ipaACMEEnabled`, `ipaACMESubCA`, `ipaACMEPrincipal`, `objectClass` | `(objectClass=ipaACMEAccount)` | `add`, `write`, `delete` |
| Read ACME Account HMAC Keys | `cn=acmeaccounts,cn=ca,$SUFFIX` | `ipaACMEHMACKey` | `(objectClass=ipaACMEAccount)` | `read`, `search`, `compare` |
| Manage ACME Challenge DNS Records | `cn=dns,$SUFFIX` | `*` | `(idnsname=_acme-challenge*)` | `add`, `write`, `delete` |

`Read ACME Account HMAC Keys` is granted only to the `ACME Server` role. It is
intentionally absent from `System: Read ACME Accounts` (a separate read-only
permission that covers all attributes *except* `ipaACMEHMACKey`) so that the
HMAC secret remains invisible to all regular LDAP readers including admins who
are not operating the ACME server.

#### Privileges

| Privilege | Contained permissions | Purpose |
|---|---|---|
| ACME Account Management | Manage ACME Accounts | Admins provisioning EAB credentials |
| ACME Certificate Request | Request Certificateâ€ , Revoke Certificateâ€ , Read ACME Account HMAC Keys | ACME server in RA proxy model |
| ACME Revocation | Revoke Certificateâ€ , Read ACME Account HMAC Keys | ACME server in HSM-backed model (no cert-request needed) |

â€  Existing IPA privilege, reused here.

#### Roles

| Role | Privileges | Typical members |
|---|---|---|
| ACME Administrator | ACME Account Management | IPA admins managing EAB accounts |
| ACME Server (RA model) | ACME Certificate Request | `acme/<hostname>` service principal |
| ACME Server (HSM-backed model) | ACME Revocation | `acme/<hostname>` service principal |

#### Host and Service Self-Service

Enrolled hosts and services can provision their own EAB accounts without admin
involvement, mirroring existing self-service patterns such as keytab retrieval.
Because EAB accounts are in a separate container (not in the host entry itself),
the self-service ACI uses the `ipaACMEPrincipal` attribute as the bind anchor:

```ldif
aci: (target="ldap:///cn=acmeaccounts,cn=ca,$SUFFIX")
     (targetattr="ipaACMEAccountID || ipaACMEHMACKey || ipaACMEEnabled ||
                  ipaACMESubCA || ipaACMEPrincipal || objectClass")
     (version 3.0; acl "Principals can manage own ACME EAB accounts";
      allow (add, write, delete)
      userattr="ipaACMEPrincipal#SELFDN";)
```

The `userattr="ipaACMEPrincipal#SELFDN"` bind rule matches when the value of
`ipaACMEPrincipal` in the target entry equals the DN of the authenticated bind
principal. For `add` operations (creating a new account entry), the IPA
framework sets `ipaACMEPrincipal` to the calling principal's DN before issuing
the LDAP add, satisfying this rule.

## Integration Models

The two integration models differ in where the sub-CA signing key resides and
where end-entity certificate signing takes place. Both models can be active
simultaneously within the same IPA deployment â€” each external ACME server is
backed by its own dedicated sub-CA and bound to a distinct service principal,
so the choice of model is made independently per ACME deployment, not globally
for the IPA installation.

### RA Proxy Model

The external ACME server acts as a Registration Authority (RA): it handles the
ACME protocol, validates EAB credentials and ACME challenges, but forwards
every CSR to IPA's `cert-request` API for signing. Dogtag signs with the
sub-CA key; the signed certificate is returned to the ACME server and
forwarded to the client. The sub-CA private key never leaves FreeIPA.

The sub-CA is created without a CSR (`ipa ca-add --acme` without `--csr-file`);
Dogtag generates the key pair and stores it on the same PKCS#11 token as the
parent CA.

This model is recommended for deployments where the ACME server has reliable
network access to an IPA CA server. Certificate issuance and revocation are
synchronous with IPA, and the full IPA audit trail is preserved.

See [RA Proxy Model design](ra-model.md) for detailed design, implementation
notes, and a test plan.

### HSM-Backed External-Key Sub-CA Model

The external ACME server generates the sub-CA key pair on its own HSM and
submits a PKCS#10 CSR to IPA. IPA signs the CSR and returns the sub-CA
certificate; the private key never leaves the ACME server. The ACME server
then signs end-entity certificates autonomously without contacting IPA at
issuance time.

The sub-CA is created with `ipa ca-add --csr-file --acme`. Dogtag signs the
submitted CSR using the `caExternalKeyCACert` profile and tracks the resulting
authority with `externalKey: true` â€” it holds no private key and performs no
signing on behalf of this authority. IPA contact is still required for EAB
validation (once per ACME account registration) and for certificate revocation.

This model is appropriate for air-gapped networks, edge deployments,
high-availability configurations where the ACME server must continue operating
during IPA maintenance windows, or deployments where security policy requires
the CA signing key to reside in a local HSM that never exports key material.

**Dogtag PKI prerequisite**: this model requires extensions to Dogtag that are
not present in the upstream release as of writing, tracked in
[dogtagpki/pki#5336](https://github.com/dogtagpki/pki/issues/5336).  In
summary, Dogtag must be extended to accept a PKCS#10 CSR at authority creation
time (`csrData` field on `POST /ca/v2/authorities`), provide a dedicated
`caExternalKeyCACert` signing profile, and handle the lifecycle of authorities
whose signing key is held externally.  See
[Dogtag PKI Requirements](external-key-model.md#dogtag-pki-requirements) in
the model design document for the full list of required changes.

See [HSM-Backed External-Key Sub-CA Model design](external-key-model.md) for
detailed design, security considerations, and a test plan.

## Choosing Between the Models

| Consideration | RA Proxy Model | HSM-Backed Model |
|---|---|---|
| Sub-CA key leaves FreeIPA | No | No (key generated on ACME server HSM) |
| Sub-CA key leaves ACME server HSM | Not applicable | No |
| IPA reachability at issuance time | Required | Not required |
| IPA reachability at account registration | Required | Required |
| IPA reachability at revocation | Required | Required |
| Revocation latency | Synchronous | Asynchronous (queue + replay) |
| Sub-CA key on HSM | Supported (IPA HSM) | Supported (ACME server HSM) |
| Deployment complexity | Lower | Moderate (CSR submission at bootstrap) |
| Recommended for | Most deployments | Air-gapped, edge, or HSM-policy sites |

In all cases, the EAB flow, LDAP schema, CLI commands, certificate profile, and
CA ACL configuration are identical. The choice of model affects only where the
sub-CA signing key resides and whether the ACME server contacts IPA at issuance
time. Both models can coexist in the same IPA deployment, each backed by a
separate ACME-dedicated sub-CA.

## Relationship to Built-in Dogtag ACME

This integration is designed to be complementary to, not a replacement for,
the built-in Dogtag ACME server. Both can be active simultaneously on an IPA
deployment. They use separate sub-CAs and separate certificate profiles. A
deployment might use the built-in ACME server for IPA-internal clients and an
external ACME server for clients in a DMZ or on edge networks.

The `acmeExternalServerCert` profile introduced here is distinct from the
built-in `acmeIPAServerCert` profile used by Dogtag ACME. In the RA proxy
model the external profile is evaluated by Dogtag at signing time using
`raCertAuth`. In the HSM-backed model it is applied locally by the ACME server,
and Dogtag is not involved at end-entity issuance time.
